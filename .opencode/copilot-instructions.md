# FHL-SpecGame — Spec-Driven Game Engine

A C++/C game + engine built entirely through spec-driven development. No hand-written implementation code — all code is generated by AI from human-authored specification documents.

## Project Structure

Something resembling:
```
specs/                     Specification documents (the source of truth)
  architecture.md          System overview, module map
  style-guide.md           Coding conventions, naming, patterns
  build/                   Build system specs
  renderer/                Graphics pipeline, shaders, sprite batching
    shaders/               Individual shader specs
  audio/                   XAudio2 setup, sound management
  input/                   Win32 input, action mapping
  math/                    Vectors, matrices, collision
  world/                   Tile maps, collision, rooms
  entities/                Player, enemies, components
  examples/                Worked example specs (reference only)
src/                       Generated implementation (output of spec pipeline)
  renderer/
  audio/
  input/
  math/
  memory/
  world/
  entities/
AGENT.md                   Full spec engine agent definition
```

## Workflow

The development workflow is: **spec → validate → generate → review**.

1. Human writes a prose spec in `specs/`
2. The spec-engine agent validates correctness, completeness, tradeoffs, and depth
3. If the spec fails any gate, the human must fix it before code is generated
4. Once all gates pass, code is generated into `src/`
5. Human reviews every line of generated code

### Pipeline Parallelism

- **Within a spec**: Gates 1–3 (Correctness, Completeness, Tradeoffs) are independent and can run in parallel after Gate 0 (Lint) passes. Gate 4 (Depth) runs after 1–2 resolve.
- **Across specs**: Multiple specs can be validated concurrently. A spec only blocks on its dependency specs' *code generation* — not on unrelated specs finishing their gates.
- **Early gate work**: If a spec's dependency hasn't finished code gen yet, run Gates 0–3 in the meantime. Only code generation requires dependency source files to exist.

## Key Concepts

- **Specs are the source of truth**, not the code. Code is a derived artifact.
- **Four validation gates**: Correctness → Completeness → Tradeoffs → Depth
- **Self-correcting specs**: Agent-verified formulas are inserted inline during gate dialogue
- **Gate persistence**: Verified sections are marked `<!-- agent-verified -->` and not re-evaluated
- **Style guide is law**: All generated code follows `specs/style-guide.md` exactly

## Language & Stack

- C/C++ (C-leaning), Win32, D3D11, XAudio2
- No engines, frameworks, or middleware
- MSBuild (.sln/.vcxproj)
